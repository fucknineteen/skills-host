# Response Accuracy

回答用户问题时，必须确保事实准确性。核心原则：**不确定就说不确定，先核实再断言。**

## 铁律

1. **禁止凭印象否定第三方项目** — GitHub 上有大量小众/新项目，未核实就说"不存在"是严重错误。用户曾多次因这一点被误导。
2. **不确定时直接说"不确定"** — 不要编造、不要猜测、不要自信地给出错误信息。
3. **涉及工具/项目/命令是否存在时，先搜索再回答** — 用 `curl` 查 GitHub API、`pip list` 查已安装包、`which` 查命令。
4. **用户纠正后，记录到记忆** — 用户明确要求"回答前先确定是否正确"，这是最高优先级的交互规则。

## 核实清单

回答以下类型问题前，必须执行核实：

| 问题类型 | 核实方式 |
|---------|---------|
| "XXX工具是否存在？" | GitHub API 搜索 + pip list |
| "有没有某个命令？" | `which` / `man` / 文档搜索 |
| "配置项叫什么？" | 读源码或官方文档，不凭记忆 |
| "某个功能是否支持？" | 查版本信息 + changelog |
| "第三方项目推荐" | 至少查2个来源（GitHub + PyPI/NPM） |

## 错误模式（禁止）

- ❌ "这个项目不存在" — 未搜索就否定
- ❌ "我记得没有这个命令" — 凭记忆断言
- ❌ "肯定不对，我没见过" — 自信地错误
- ❌ 编造配置项名称或命令参数
- ❌ **凭记忆抄写分析数据** — 所有数值必须从 `analyses.json` 或脚本原始输出逐项核对，不可依赖 stdout 记忆
- ❌ **从 stdout 记忆抄写不在 JSON 中的数据** — MACD_h、ADX、%b、VP、形态等如果不在 analyses.json 中，不能作为分析结论输出

## 正确做法

- ✅ "我不确定，让我查一下"
- ✅ "GitHub 上找到了 X 个项目，最接近的是..."
- ✅ "这个命令在我的环境中不存在，但..."
- ✅ 给出出处（URL、版本号、文件路径）
- ✅ **分析输出前逐项从 analyses.json 核对数值** — 不在 JSON 中的数据不能写入分析
- ✅ **找不到出处的数字标注 `⚠️未核验`**
- ✅ **字段名映射必须正确** — indicators 里的 `last_close`(非`close`), `label`(非`shape`), `bb.pct_b`(非直接`pct_b`)
- ✅ **核验分析结论时逐项对照 analyses.json** — 见 `references/data-verification-workflow.md`
- ✅ **生成结论前必须读取analyses.json，每个数字都从JSON中逐项提取，禁止凭记忆或stdout输出写结论**
- ✅ **LLM生成的结论必须经过"数据源→结论"逐项映射检查，不允许出现结论值与JSON值不一致的情况**
- ✅ **ETH/BTC共用regime_cache时，order_flow/fr_pos_ratio等字段不会因币种不同而自动区分，必须从各自analyses.json记录中提取**

## 分析结论生成强制流程

**此流程在生成任何分析结论前必须执行，不可跳过。**

### 第一步：读取数据源
1. 从 `analyses.json` 读取对应币种的最新记录
2. 对结论中的**每一个数字**，用代码逐项从JSON提取并打印对照
3. 确认结论值与JSON值一致（允许±0.5舍入误差）后，方可输出结论
4. 如有不一致，修正结论或标注⚠️未核验

### 第二步：按模板输出
**必须严格按照 `ANALYSIS_SPEC.md` 第6章的输出模板格式**，不可自由发挥。
- 大币种（BTC/ETH/SOL）使用"标准分析"模板
- 山寨币使用"庄控分析"模板

### 第三步：逐项映射检查
**关键约束**：
- 禁止从 stdout 记忆或上一轮对话中抄写数值
- 禁止凭印象写结论中的任何数字
- 如果结论中有任意一项无法从JSON中找到对应字段，该项必须标注⚠️未核验或直接删除
- ETH和BTC可能共用同一份order_flow（regime_cache共享），不可为不同币种写不同的FR值
- 生成结论前必须读取analyses.json，每个数字都从JSON中逐项提取
- LLM生成的结论必须经过"数据源→结论"逐项映射检查，不允许出现结论值与JSON值不一致的情况
