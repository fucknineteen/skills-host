# 营销内容验证

## 何时加载

用户提到视频、文章、文案中引用的工具/产品/服务，要求核实真实性时加载。常见触发词：「视频里说的XX是真的吗」「这些工具哪个有用」「验证一下」「有没有虚构的」。

---

## 验证流程

### Step 1: 提取所有工具名称

列出文案/视频中提到的全部工具名，去重。

### Step 2: 逐个核实存在性

**首选方法 — GitHub API 精确查询：**

```bash
curl -s https://api.github.com/repos/<owner>/<repo> | head -5
```

- 返回 `404` → 仓库不存在
- 返回 `Not Found` → 大概率虚构
- 返回 JSON 含 `full_name` → 真实存在

**备选方法 — GitHub API 搜索：**

```bash
curl -s "https://api.github.com/search/repositories?q=<关键词>&sort=stars"
```

- 注意：搜索结果可能有同名但无关的项目
- 看 `total_count` 和 `items` 里的 `full_name` 是否匹配
- 看 `description` 是否对应声称的功能

**辅助方法 — pip 包验证（Python工具）：**

```bash
pip install <package-name> --dry-run
```

- 注意：有些名字相同但功能完全不同（如 `jina` 是NLP框架，不是Jina Reader）
- 注意：有些工具只有API/SDK没有pip包

**辅助方法 — 官网/API文档验证：**

```bash
curl -sI https://<domain>.com | head -3
```

- 404/503 → 可能已下线
- 正常响应 → 继续验证

### Step 3: 交叉验证

对每个工具记录：

| 状态 | 含义 |
|------|------|
| ✅ 真实存在 | GitHub API 200 + 描述匹配 |
| ⚠️ 存在但无关 | GitHub 搜到同名但功能不对 |
| ⚠️ SaaS/闭源 | 有API但无开源repo（如Tavily） |
| ❌ 虚构 | 所有渠道都搜不到 |
| ❓ 需进一步验证 | 信息不足，无法判断 |

### Step 4: 评估实用性（对用户）

对每个真实存在的工具，判断：
- 是否与用户当前工作流相关
- 是否有更好的替代方案
- 安装/配置成本 vs 收益

---

## 常见虚构模式

1. **完全捏造** — GitHub/pip/官网全搜不到（如RTK、Talkscale）
2. **拼写错误** — 名字接近真实工具但不同（如HTTS vs TTS）
3. **功能夸大** — 存在但声称的能力远超实际
4. **过时信息** — 曾经存在但已下架/改名/停更
5. **山寨混淆** — 名字模仿知名项目

---

## 输出格式

按优先级排序输出：

```
✅ 真实存在的工具：
1. [名称] — [简述]（⭐[星数]）— [对用户是否有用]

❌ 不存在/虚构的工具：
2. [名称] — [搜不到/404] — [可能是虚构/拼写错误]

⚠️ 需注意：
3. [名称] — [存在但无关/功能不符/需付费]
```

## 注意事项

- **不要假设** — 每个工具必须实际查询，不能凭印象判断
- **注意大小写** — GitHub 仓库名区分大小写
- **注意域名** — 有些工具有独立官网但不是GitHub项目
- **注意API vs 开源** — 有些是纯SaaS服务（如Tavily），没有开源代码
- **注意版本** — 有些工具已改名/迁移到新组织
- 对营销内容保持怀疑 — 视频/文章有动机夸大，交叉验证是必须的
- **实用性评估必须基于用户实际工作流构成** — 不要只看工具本身功能，要算它覆盖用户日常操作的百分比（详见 `references/rtk-compression-scope.md`）

## Pitfall: 实用性评估要基于用户实际工作流构成

不要只说「有用/没用」，要分析用户实际使用场景的构成比例。例如：

- 用户的交易工作流 90% 是 Python 脚本（`execute_code`），9% 是文件读写（`write_file`/`read_file`），1% 是终端命令（`terminal`）
- RTK 只压缩终端命令输出，对用户来说节省不到 2% 的 token
- 结论：「RTK 对你作用很小，别装」

同理，Dashboard 工具（hermes-ui、hermes-workspace、mission-control）虽然有 token 统计功能，但部署成本远大于收益，因为用户主要通过 Telegram 交互。

**评估公式**：安装/配置成本 vs 实际使用频率 × 收益。如果用户 99% 的场景已覆盖，新工具不值得装。

## 工具实用性评估方法论（2026-06-19 新增）

**不要只看工具功能，要看它覆盖用户实际工作流的百分比。**

### 评估框架

对每个推荐工具，问三个问题：

1. **它压缩/加速的是什么？** — 定位工具作用的具体环节
2. **用户在这个环节花多少时间/成本？** — 量化实际影响
3. **安装/配置/维护成本 vs 收益？** — 净收益是否为正

### 实例：RTK 终端压缩

- 工具作用：压缩终端命令（`git log`、`ls -la`、`pytest`）输出
- 用户工作流构成：90% Python脚本(`execute_code`) + 9% 文件读写(`write_file`/`read_file`) + 1% 终端命令
- RTK 覆盖范围：仅1%的终端命令输出
- 结论：节省不到2%的token，别装

### 实例：Dashboard/仪表盘

- 工具作用：Web界面展示token用量、成本、会话管理
- 用户交互方式：主要通过Telegram，不需要Web界面
- 安装成本：部署+维护+学习曲线
- 结论：可视化收益 < 部署成本，别装

### 实例：SkillClaw

- 工具作用：自动去重/进化/改进技能库
- 用户技能库规模：约20个精心维护的技能
- 用户痛点：技能库不会乱，有人（我）协助管理
- 结论：解决的是"几百个技能堆积如山"的问题，用户不存在这个问题，别装

### 评估公式

```
净收益 = (实际使用频率 × 单次收益) - 安装成本 - 维护成本
```

- 频率高、单次收益大、成本低 → 装
- 频率低、收益小、成本高 → 别装
- 频率中等但有更好替代 → 别装

### 关键原则

- **用户的工作流构成是评估的基础** — 不知道工作流就无法评估
- **不要只看功能列表** — 功能多不等于对你有用
- **不要假设用户需要** — 每次评估都要基于实际工作流数据
- **宁可少装，不要乱装** — 多一个工具就多一个维护负担
