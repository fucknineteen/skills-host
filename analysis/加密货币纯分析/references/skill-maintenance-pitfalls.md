# 技能文档维护陷阱

> 来源：2026-06-20 快讯集成 + 全量审计
> 适用：任何修改 `加密货币纯分析` 或 `社交动态发布` 技能时的代码审查

## 常见问题模式

### 1. 字段计数遗忘

新增字段后（如 `flash_news`），SKILL.md 中多处出现硬编码计数（如"五个社交字段"），容易只改一处漏另一处。

**检查点**：
- 全文搜索计数关键词（"N个"、"N条"、"N步"）
- 确保所有出现位置一致

**本次实例**：
- `crypto §2.3`：说"五个" → 实际六
- `social §9`：说"五个" → 实际六
- `social §2.2`：已正确说"六个"（先改的这里）

### 2. 记录字典漏字段

`analyses.json` 和 `social_analyses.json` 的记录 dict 是各自独立构建的，新增字段时必须两处都加。

**本次实例**：
- `analysis_template.py` main() 记录构建：加了 `flash_news` ✓
- `publish_social.py` 记录构建：漏了 `flash_news` ✗（已修复）

**检查点**：
- 搜索两个文件中所有的 `record = {` 或 `merged.append({`
- 确认新增字段在两个 dict 中都存在

### 3. 过时的「已知问题」

修复 bug 后，SKILL.md 的"已知问题"/"已知坑"章节容易忘记同步更新。

**本次实例**：
- §6 "generate_social_draft 死代码" → 代码已删，文档还在说
- §10.5 "快讯/新闻未集成" → 已集成，文档未更新

**检查点**：
- 每次修 bug 后，全文搜索 bug 关键词看文档是否有残留
- 修完后立即更新对应章节

### 4. 死代码清理副作用

删除函数后，SKILL.md 中引用行号的说明会偏移。如 `L1297-1320` 在删除 143 行后变成 `L1154-1177`。

**检查点**：
- 删除/插入超过 50 行代码后，全文搜索 `L\d` 引用
- 这些引用是易碎的，优先用函数名替代行号

### 5. 双文件同名函数

`analysis_template.py` 和 `_social_publish.py` 都有 `generate_social_draft()`，容易误改错版本。

**规则**：
- `publish_social.py` 始终从 `_social_publish` import → 只改 `_social_publish.py`
- `analysis_template.py` 中的同名函数是死代码 → 应删除，不要改

**本次实例**：
- 首次添加 📰 行时，两处都改了（因为不确定哪个在用）
- 审计确认 `analysis_template.py` 版本是死代码后，直接删除
