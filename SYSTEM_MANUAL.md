## 3. 工作流（5个核心）

### 3A. 加密货币纯技术分析

- **入口**: `python3 analysis_template.py BTC ETH`
- **覆盖币种**: BTC / ETH / SOL / DOGE
- **流程步骤**:
  1. 同步 K 线数据
  2. 计算技术指标
  3. 检测威科夫分布
  4. 检测 K 线形态
  5. 成交量分布 (VP)
  6. 订单流分析
  7. 道氏理论分析
  8. 共振评分
  9. 宏观环境
  10. 财经日历
  11. 金十快讯
  12. 写入 `analyses.json`
- **输出模板**:
  - 大币种（BTC/ETH）→ 标准技术分析模板
  - 山寨币（SOL/DOGE）→ 庄控分析模板
- **严格规则**:
  - ❌ **禁止凭记忆或 stdout 输出写分析数字**
  - 所有数值必须从 `analyses.json` 逐项提取并核对

#### 字段映射表

| 分析用语 | JSON 路径 |
|----------|-----------|
| 现价 | `indicators.1H.last_close` |
| RSI | `indicators.4H.rsi` |
| MACD 柱 (MACD_h) | `indicators.4H.macd_h` |
| %b (布林带位置) | `indicators.4H.bb.pct_b` |
| K线形态标签 | `indicators.4H.label` |
| VP (成交量分布) | `session_vp.POC` / `session_vp.VAH` / `session_vp.VAL` |
| 资金费率 (FR) | `order_flow.funding_rate_pct` |
| 恐惧贪婪指数 (FG) | `macro_external.fg_actual` |

### 3B. 社交动态发布

- **入口**: `python3 publish_social.py BTC ETH`
- **7步流水线**:
  1. 同步 K 线数据
  2. 分析当前行情
  3. 复盘上一条动态
  4. 撰写文案
  5. 数据核验
  6. 用户审核
  7. 配图 + 保存

- **文案模板**: 社交动态模板 v5.1（人类自然风格，不念经）

- **关键约束**:
  - 观望方向时，🎯 行输出 **"空仓等风"**（2026-06-20 确认）
  - RR（盈亏比）< 1.5 时标 ⚠️
  - 每段至少包含 **2-3 个精确数字**
  - **去废词**: 删除"建议"、"可能"、"或许"等模糊词汇

- **核验**: `verify_social_post.py` v4 架构（数据源驱动），核验不通过 `exit 1` 阻断保存

- **禁止发布窗口**:
  - FOMC 前 48 小时
  - CPI / 非农前 12 小时
  - 4★ 及以上事件前 6 小时

- **配图**: `gen_charts.py` 提供 4 种风格自动选择

- **复盘**: `review_last_post.py` — **一次性复盘**，与三层复盘引擎（process_reviews.py）是独立的两套系统

### 3D. 三层复盘引擎

- **入口**: `process_reviews.py`，由 cron 在每小时 `:03` 触发
- **三层结构**:
  1. **6小时快判** — 初步判定方向对错
  2. **12小时深复盘** — 深度审视分析逻辑
  3. **72小时终局** — 最终结论

- **教训提取逻辑**:
  - `total ≤ 1` 时触发教训提取
  - 即：+2 或 +3 的结果才跳过，不提取教训
  - `total = +1`、`0`、`-1` 及以下都需要提取教训

- **教训注入**:
  - `inject_lessons.py`（cron `:05`）→ 写入 `.lesson_context.txt`
  - 分析前**强制加载**该文件

- **与社交动态复盘的关系**: 两套独立系统，互不影响
  - 三层复盘 (`process_reviews.py`)：持续运行，覆盖所有分析
  - 社交复盘 (`review_last_post.py`)：一次性，仅针对最新一条社交动态

## 4. 编码规范

### 4.1 代码审查原则（Karpathy 四大原则）

1. **先思考再动手** — 理解全貌后再修改
2. **简洁优先** — 能用简单方案不用复杂方案
3. **手术式修改** — 精准定位，最小化改动范围
4. **目标驱动** — 每次修改对应明确的修复目标

### 4.2 修复流程

```
全量审计 → 按严重度分组报告（Bug / 逻辑错误 / 代码气味）→ 用户说"全修" → 逐项修复
```

### 4.3 修复指令规则

- 用户说 **"修复"** = 全部修复，不给选择题、不问优先级
- 用户说 **"全修"** = 全部修复，同上
- **审查时**只报告事实和严重度，**不擅自删除别人的死代码**

### 4.4 文件安全

- ❌ **禁止主动删除、移动或重命名文件**
- 修改任何文件前，必须先向用户确认
- 如果确需操作文件，等待用户明确指令

## 5. 时区规则

> **极其重要 — 违反此规则会导致时间计算全部错误**

### 5.1 基本规则

- 所有时间统一使用 **BJ (UTC+8)**
- 适用范围：
  - 蜡烛标签时间
  - 收盘时间
  - 数据时间戳
  - 宏观事件时间（CPI、非农、FOMC）

### 5.2 Python 时区正确写法

**✅ 唯一正确写法**（二选一）:

```python
# 写法一：从 UTC 转换
NOW_UTC.astimezone(BJT)

# 写法二：直接创建 BJ 时间
datetime.now(BJT)
```

**❌ 绝对禁止**:

```python
# 错误！只改时钟数字，不改变 tzinfo 对象
# Python 做减法时会把 BJ 时间当成 UTC 来处理
NOW_UTC + timedelta(hours=8)
```

### 5.3 时区定义来源

- `BJT` 时区在 `_shared.py` 中定义
- 所有工作流**统一引用**该定义，不得各自定义

1. **禁止凭印象断言**第三方项目、工具、命令的存在或不存在

## 7. 仓位/方向逻辑

### 7.1 方向字段

- `position` 字段取值：**偏多** / **偏空** / **观望（等确认）**

### 7.2 共振评分体系

```
共振评分 = RS_I_4H(±1) + MACD_h_4H(±1) + %b_1H(±1)
```

判定规则：
- `|score| ≥ 2` → 偏强（正）或偏弱（负）
- `|score| < 2` → 分歧，观望

### 7.3 near_top 捷径（2026-06-20 新增）

同时满足以下三个条件 → 直接判定**偏空**：
1. `RSI_1d > 67`
2. 1D K 线下降
3. `MACD_4h < 0`

### 7.4 V 反保护

出现以下任一情况 → **降级为观望**：
- `near_bottom == True`
- 最近 8 根 4H K 线反弹幅度 > 3%

### 7.5 SL/TP 方向感知

- 做多：止损 (SL) 放在低位下方
- 做空：止损 (SL) 放在高位上方

### 7.6 entry 价格

- 使用 `ticker.last`（现价），**不用日线收盘价**

### 7.7 MACD 使用

- **MACD_4H** 用于方向判定
- 原因：滤掉日内噪声，对齐 6h-72h 持仓周期
