# 文件安全守则

> 每次对话必加载。优先级高于所有其他 skill。

## ⛔ 禁止操作（必须用户确认）

1. **禁止主动删除文件** — 包括 `rm`、`unlink`、`os.remove`、`shutil.rmtree` 等
2. **禁止主动移动/重命名文件** — `mv`、`os.rename`、`shutil.move`
3. **禁止覆盖写入** — 修改已有文件内容前必须先备份或用户确认

## 确认流程

执行上述任何操作前：
1. 列出所有将受影响文件的**绝对路径**
2. 说明操作原因
3. 等待用户明确同意（「可以」「行」「删吧」等）

## 唯一例外

用户指令中明确说出「删除 xxx 文件」「把 xxx 删掉」时可直接执行，无需二次确认。

## 🏷️ 工作流命名规范

- **新建工作流必须用中文命名**，禁止英文名（如 `crypto-analysis-workflow` ❌ → `加密货币纯分析` ✅）
- 命名来源：取技能 `description` 的核心功能提炼，4-6字为宜
- 系统自带的英文技能（如 `skill-management`、`apple-notes`）不得改名

## 🔄 技能改名安全检查（改名操作不视为"移动"，但需走此流程）

改名前逐项确认：
1. `hermes cronjob list` — 无 cron 引用该技能名
2. `grep` 所有 SKILL.md — 无交叉引用英文名
3. `grep` 所有脚本 — 无 `import` 或字符串引用该技能名

改名步骤：
```
① mv 旧英文目录 新中文目录
② sed -i 's/name: 英文名/name: 中文名/' 新目录/SKILL.md
③ sed -i SKILL.md 全文替换交叉引用
④ 更新 auto_push_github.sh（如有脚本路径引用）
⑤ 同步到 mirror 仓库 + git push
⑥ skills_list 验证新名可见
```

## 🔍 工作流文档审计模式

当修改脚本后，检查 SKILL.md 是否同步：

1. **字段计数** — 如「五个社交字段」实际已是六个 → 更新
2. **死代码引用** — 如 SKILL.md 说「L1544 存在 generate_social_draft 副本」但代码已删 → 删除
3. **过时章节** — 如「快讯未集成」但已集成 → 更新
4. **行号引用** — SKILL.md 中的行号随代码改动偏移，仅作参考不要视为精确
